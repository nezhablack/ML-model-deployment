## 1. Постановка A/B‑теста

Цель: сравнить две версии модели прогнозирования дефолта по кредитным картам и выбрать модель для продакшена.

Варианты:

- **Группа A (контрольная)** - текущая модель `model_v1` (LogisticRegression) с действующим порогом.
- **Группа B (тестовая)** - альтернативная модель `model_v2` (например, Gradient Boosting или XGBoost).

Разделение трафика:

- пользователи/заявки случайно распределяются между группами A и B;
- целевой вариант - **50/50**, либо более осторожно 80/20 на старте;
- ключ для сплита - стабильный идентификатор `ID`, чтобы один и тот же клиент всегда попадал в одну группу.

Продолжительность:

- до накопления статистически значимого числа заявок и дефолтов (минимум один платежный цикл, в зависимости от потока заявок и частоты дефолтов).


## 2. Метрики

### 2.1. ML‑метрики

Основная метрика:

- **F1‑score для класса дефолта** - как баланс Precision и Recall по ненадежным клиентам.

Дополнительные метрики:

- **Recall** для класса дефолта, если важнее выявить максимум дефолтов и не пропускать рисковых клиентов,
- **Precision** для класса дефолта, если важнее не блокировать слишком много надежных клиентов.

Обоснование:

- F1 используется как основная, потому что она учитывает и пропуски дефолтов (Recall), и ложные срабатывания (Precision);
- выбор Precision или Recall как дополнительной метрики зависит от бизнес‑приоритета (агрессивное снижение риска или более мягкая политика одобрений).

### 2.2. Бизнес‑метрики

- **Expected Loss (EL)** - оценка ожидаемых потерь по портфелю на основе модели вероятности дефолта и параметров кредита;
- **Approval Rate при фиксированном уровне риска** - доля одобренных заявок при выбранном пороге вероятности дефолта и допустимом уровне дефолтов.

Обе метрики считаются отдельно по группе A и по группе B.



## 3. Статистический анализ

### 3.1. Сравнение метрик A и B

Для сравнения F1 (и других метрик) между группами A и B:

- используем **z‑test** для разницы долей (или бутстрэп‑оценку доверительного интервала разницы F1 между моделями);
- строим **95% доверительные интервалы** для разницы метрик (F1_B − F1_A, Precision_B − Precision_A, Recall_B − Recall_A, а также для разницы долей дефолтов/одобрений).

### 3.2. Критерий успешности

Модель B считается лучше A и может быть принята в качестве новой продакшен‑модели, если выполняются все условия:

1. F1_B >= F1_A, и разница статистически значима: p‑value по z‑test < 0.05 и 95% доверительный интервал разницы (F1_B - F1_A) не включает ноль.
2. Дополнительная метрика (Precision или Recall) не ухудшается существенно. При этом допустимое снижение заранее фиксируется.
3. Бизнес‑метрики не ухудшаются:
   - Expected Loss не растет,
   - Approval Rate остается в допустимых пределах при том же уровне риска.
4. Latency и стабильность API (ошибки 5xx, падения контейнеров) не хуже, чем у модели A.



## 4. Связь с архитектурой сервиса

- Разделение трафика по `ID` реализуется на уровне веб‑сервиса: часть запросов направляется на модель A, часть - на модель B. В монолитном Flask‑сервисе это может быть выбор между двумя загруженными объектами модели внутри одного процесса.
- Логи запросов (`model_version`, `prediction`, `probability`, фактический исход дефолта) используются для последующего анализа A/B‑теста.
- В будущем возможен перенос маршрутизации и логирования на внешний слой (например, NGINX + отдельный лог‑обработчик или брокер сообщений).

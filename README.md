# ML-model-deployment
Разработка и внедрение сервиса прогнозирования дефолта по кредитным картам

Проект представляет собой полноценный ML-сервис для прогнозирования дефолта по кредитным картам на основе датасета Default of Credit Card Clients (UCI / Kaggle).  
Проект демонстрирует полный цикл внедрения модели: обучение, сохранение, REST API, Docker-образ и концепцию A/B‑тестирования.

## 1. Цель и постановка задачи

**Цель:** построить и задеплоить веб-сервис, который принимает признаки клиента кредитной карты и возвращает прогноз дефолта в следующем месяце и вероятность дефолта.

- **Предметная область:** кредитный скоринг, финансы.
- **Целевая переменная:** `default.payment.next.month` (переименован в `target`).
- **Датасет:** [Default of Credit Card Clients Dataset (Kaggle)](https://www.kaggle.com/datasets/uciml/default-of-credit-card-clients-dataset).

## 2. Данные

- Источник: Kaggle, датасет кредитных карт на Тайване.
- Объекты: клиенты с историей платежей и демографией.
- Таргет: бинарный признак дефолта в следующем месяце.
  При обучении:
  - таргет: `default.payment.next.month` - `target`;
  - признак `ID` удаляется;
  - остальные столбцы используются как признаки.

## 3. Архитектура сервиса

### 3.1. Общая схема

- **Монолитный Flask-сервис**, который:
  - при старте загружает модель из `models/model_v1.pkl`;
  - предоставляет два HTTP эндпоинта:
    - `GET /health` - health‑check;
    - `POST /predict` - инференс.
- Модель - sklearn pipeline (`StandardScaler` + `LogisticRegression`) и подобранный порог классификации, сохранённые через `joblib`.

Выбор **монолита** для учебного проекта обоснован так:

- одна модель, один сервис, один человек в команде, микросервисы добавили бы лишнюю сложность (отдельные деплои, межсервисные вызовы, мониторинг)
- монолит проще запускать локально и в Docker, проще проверять по чек‑листу
- структура кода подготовлена так, чтобы модель, API и окружение можно было при необходимости выделить в отдельные сервисы.

В production монолит можно было бы разделить на:
- отдельный сервис инференса,
- сервис логирования/мониторинга,
- сервис для асинхронных задач (batch‑предсказания, обучение второй модели).

### 3.2. Логирование и мониторинг

- В production эти логи могут собираться агентами (например, Promtail или встроенным Docker‑драйвером логирования) и отправляться в Loki, а затем визуализироваться в Grafana. Такой стек позволяет:
- мониторить доступность и задержки ответа API,
- анализировать ошибки и аномалии в логах,
- строить дашборды для отслеживания результатов A/B‑тестов (фильтруя записи по полю `model_version` и другим меткам).

### 3.3. Message broker (концепт RabbitMQ)

В рамках проекта брокер сообщений не реализован, но концептуально RabbitMQ мог бы использоваться для:

- асинхронной обработки batch‑предсказаний (передавать задачи в очередь и обрабатывать воркерами);
- асинхронного логирования (слать события о запросах/предсказаниях в отдельный лог-сервис);
- поддержки A/B‑тестирования (слать события с версией модели и фактическими исходами).

## 4. Модель и MLOps-концепты

### 4.1. Модель v1

Файл: `models/model_v1.pkl`  
Содержимое:

```python
{
  "pipeline": sklearn.pipeline.Pipeline,  
  "threshold": float                   
}
```

Обучение:

- модель: `LogisticRegression(max_iter=1000, class_weight="balanced", random_state=42)`;
- pipeline: `StandardScaler` + `LogisticRegression`;
- разбиение: `train_test_split(test_size=0.2, stratify=y, random_state=42)`;
- метрика для подбора порога: F1 для класса дефолта;
- сохранение: `joblib.dump({"pipeline": pipeline, "threshold": best_t}, "models/model_v1.pkl")`.

### 4.2. DVC и MLflow 

В проекте нет обязательной интеграции с DVC/MLflow, но:

- Data Version Control использовался бы для:
  - версионирования датасетов (`data/`),
  - хранения артефактов моделей (`models/model_v1.pkl`),
  - воспроизводимости экспериментов (связь 'код - данные - модель').

- **MLflow**  
  Использовался бы для:
  - логирования экспериментов (параметры модели, метрики F1/Precision/Recall, пороги);
  - трекинга версий моделей (model registry);
  - выбора лучшей модели для production и A/B‑тестирования.

## 5. API

### 5.1. Endpoints

- `GET /health` — проверка работоспособности сервиса;
- `POST /predict` — инференс по одному клиенту.

### 5.2. Health-check

```bash
GET /health
```

Пример (PowerShell):

```powershell
Invoke-WebRequest -Uri "http://127.0.0.1:5000/health" -Method GET
```

Ответ:

```json
{"status": "healthy"}
```

### 5.3. Predict

**Request**

```http
POST /predict
Content-Type: application/json

{
  "features": [f1, f2, ..., fN]
}
```

`features` — список чисел в том же порядке, в котором признаки подавались на вход модели при обучении.

**Пример (PowerShell)**

```powershell
$body = @{
    features = @(20000,2,2,1,24,2,2,0,0,0,0,3913,3102,689,0,0,0,0,689,0,0,0,0)
} | ConvertTo-Json

(Invoke-WebRequest `
  -Uri "http://127.0.0.1:5000/predict" `
  -Method POST `
  -ContentType "application/json" `
  -Body $body).Content
```

**Пример ответа**

```json
{
  "prediction": 1,
  "probability": 0.7923,
  "threshold": 0.6
}
```

## 8. Бизнес-метрики

Помимо F1/Precision/Recall для модели, проект ориентирован на бизнес‑метрики:

1. **Ожидаемые потери (Expected Loss)**  
   \[
   \text{EL} = \sum_i PD_i \times EAD_i \times LGD_i
   \]  
   где:  
   `PD` - вероятность дефолта от модели,  
   `EAD` - экспозиция по кредиту,  
   `LGD` - доля потерь при дефолте.  
   Цель - минимизировать ожидаемые потери при сохранении приемлемого уровня одобрений.

2. **Approval rate при фиксированном уровне риска**  
   Доля одобренных заявок при заданном пороге PD и допустимом уровне дефолтов. Используется, чтобы при переходе на новую модель сохранять (или увеличивать) долю одобрений при том же уровне кредитного риска, а не просто не выдавать кредиты.

## 9. A/B‑тестирование

В проекте предусмотрена концепция A/B‑теста между:

- **v1** — текущая production‑модель (`model_v1.pkl`),
- **v2** — улучшенная модель (альтернативный алгоритм или другой threshold).

План A/B‑теста вынесен в отдельный файл `AB_TEST_PLAN.md`.

Кратко:

- трафик делится случайно 50/50 по `client_id`
- основная метрика: F1 для класса дефолта
- вспомогательная: Precision или Recall, в зависимости от приоритетов бизнеса
- для анализа используется z‑test или бутстрэп‑интервалы для разницы метрик
- критерий успеха: значимое улучшение основной метрики без критического ухудшения вспомогательной.

## 10. Известные ограничения

- Модель сохранена с использованием scikit‑learn, что делает `.pkl` файлово- и версионно‑зависимым артефактом.  
- Для реального production-деплоя рекомендуется:
  - фиксировать версии зависимостей в `requirements.txt`;
  - при обновлении sklearn переобучать и пересохранять модель в новой версии.
